Programming FSM onto hardware

The goal of this section is that, after learning the abstraction of FSM, we want to make a physical machine that can conform to the functional specification of our designed FSM. For any FSM, we can say that in general we need iii input bits, sss state bits, and ooo output bits.

  1. By creating a ROM: use a demultiplexer, with each output bit soldered to conform to the truth table

FSM Limitations

Some problems cannot be computed using FSMs, so FSMs alone is not enough to be an ultimate computing device. A classic example that cannot be computed using FSM is \textit{parenthesis checker}. ~\

~\

Reason: it requires arbitrarily many states because you can have as many left parenthesis as you want and you need to keep count of all left parenthesis to check if it is balanced with the right parenthesis. By definition, an FSM needs a FINITE amount of states.

Turing Machines

The Turing model solves the \textit{finite} problem of the FSM, so basically it solves the run-out-of-state problem in FSMs. See diagram below of arbitrary FSM implemented as Turing machine:

\includepic{0.3}{S2}

Explanation:

  1. The idea is to imagine you have a physical machine (like a typewriter machine or something), called the Turing Machine, that can take input of an infinitely long “tape” (its basically an array) that you can write at (‘0’ or ‘1’ , or anything), and it has a functional specification table that describes its behavior

  2. And then you have the pointer in the machine (black arrow) that represent your current state location

  3. You can move the tape left and right to shift the arrow, or \textit{you can move the arrow left and right}. Note that in order to move the arrow right, you need to equivalently move the tape left.

  4. Theres a HALT state where you reach the end-state

  5. So at Figure 2, assume that the start state is S0S0S0, and as shown the black arrow is at 1, so its equivalent as saying that the input is 1

  6. Then look at the functional specification table, and decide where to go next.

  7. From the table, if current state is S0S0S0 and input is 111, you write 1 and then move the tape (depends on the specification – since in theory you can move the arrow or tape, whichever that is easier to think) to the right

  8. \textit{It is important to write first, and then move the arrow}, not the other way around

  9. Repeat until you meet a HALT as next state

  10. Note: in practice, we move the TAPE, not the arrow. So a RIGHT movement of the arrow means moving the tape to the LEFT. To make it easier to learn the TM functionality for the first time, we can imagine that we move the arrow instead. The movement of the tape and the arrow is the opposite of one another

In this section we are going to go through a an example of how turing machine can be used as a machine that “reads some integer”, and we can also build another turing machine that takes this integer (the output of another turing machine) as an input and perform some function on it. This is just one of the many many functionalities that a Turing Machine can do.

See the diagram below. The arrow represents the beginning of an arbitrary FSM called FSM347FSM_{347}FSM347​ (the name doesnt mean anything, its just some naming) that operates on a Tape number 51. To read that integer 51 on the tape (so that you know which tape you’re at), you can do:

  1. Take the number at the arrow head as the least significant bit, in this case thats 1.

  2. Then you can move on either to the left or right depending on the number that you encode. Lets say in this case you (the machine) move to the right (or equivalently move the tape to the left) to b1b_1b1​ position and append it as the second least significant bit, so thats 11

  3. After this you should move to the left to reach b2b_2b2​ (because previously you chose to move to the right), and append as the third least significant bit: 011

  4. Now you need to move to b3b_3b3​, and repeat to append as the fourth least significant bit: 0011

  5. Oscillate back to b4b_4b4​, 10011

  6. Oscillate to the right to b5b_5b5​, 110011

  7. Continuing until b8b_8b8​ we have 000110011 and so on

  8. So we can end up with finite integers with potentially infinite leading trail of zeroes to represent our integer

  9. In this case, 000110011 is 51

Church’s Thesis and Computable Function

The Church’s Thesis: Every discrete function computable by ANY realisable machine is computable by some Turing machine.

~\

~\

So, by definition:

[

f(x) \text{ is computable for some k, all x if } f(x) = T_x[x] = f_k(x)

]

[

\textit{ (equivalently, f(x) is computable if it is computable by Turing Machine)}

Universal Function

[

U(k,j) = T_k[j]

Explanation:

  1. The universal function is a model of general purpose computer

  2. Imagine a device with many physical Turing machines inside it, labeled as T1,T2,...Tk...T_1, T_2, ... T_k...T1​,T2​,...Tk​.... This will be such a big device.

  3. Remember, each Turing machine has its functional specification table, and the data “tape”

  4. For example, you have Turing machines like Figure 5 below in your lab:

\includepic{0.2}{S4}

  1. So back to the universal function above, kkk is the input, basically a program to tell us which of computable functions TkT_kTk​ we want to compute,

  2. jjj is the data tape TkT_kTk​ is going to perform at

  3. TuT_uTu​, our universal machine, interprets the data, kkk and jjj emulating its program kkk process on the data, in other words, to emulate the behavior of the TkT_kTk​ (kth Turing Machine) to operate on the data jjj

  4. So with TuT_uTu​, we no longer need to make so many physical Turing Machines to perform the necessary function, because TuT_uTu​ can emulate the behavior of any Turing Machines

  5. TuT_uTu​ is basically our computer, it can read our codes and perform it just as intended. We can just write the code kkk, and execute it on TuT_uTu​. For example, we want to multiply a data, factorize them, and then sort them. We no longer need to build three different physical system to multiply, factorize, and split. We just need our one computer to do all the three different tasks.

The Computer Science Revolution

Imagine building a machine whose input and output is another physical machine, we need combine and compile them to make a new physical machine with different specs. For example, we want to make a physical hardware that can perform physical sorting using a method like insertion sort, or merge sort. This is very complex to do. We need to replace these “hardwares” with a coded description of that piece of hardware (which is software), then its easy to cut, change, mix, and modify them around. This is the universal Turing Machine: a coded description of a piece of hardware. This means to move from hardware paradigm into data paradigm. People are no longer spending their time sitting in workshops making physical systems but they are now sitting in front of computers writing codes. Programs can easily input and output other programs.
